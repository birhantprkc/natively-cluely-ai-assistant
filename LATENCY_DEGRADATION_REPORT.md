# Latency Degradation Report — long-session manual testing (2026-06-12)

## Reported symptoms (user manual testing, ~200 questions)

| symptom | measured (user log sample) |
|---|---|
| first-useful token avg | ~1755ms |
| first-useful median | ~1499ms |
| first-useful p95 | ~3316ms |
| first-useful max | ~4781ms |
| one provider_timeout | at ~7003ms after `processQuestion` took ~6906ms |
| app feel | "lagging/hanging after ~50 questions" |

## Root causes found (with fixes)

### 1. `processQuestion` could run a LIVE company-research call in the hot path — ~6.9s
`premium/electron/knowledge/KnowledgeOrchestrator.ts` — when no cached dossier
existed, `researchCompany()` ran **synchronously before the provider was even
dispatched**. That single call explains the 6906ms processQuestion → 7003ms
provider_timeout → fallback chain in the log.

**Fix**: live research is now budget-bounded to **2s** in the answer path
(`Promise.race`); on timeout the answer proceeds without the dossier and the
research continues in the background so the *next* question hits the DB cache.

### 2. Gap pivot scripts injected for ANY question mentioning a gap skill
The log's repeated "Injecting 1 pivot script(s) for detected gap skills" fired
on unrelated questions (any candidate-directed turn whose text contained a gap
skill token), adding hundreds of chars to every prompt and steering off-topic
answers toward pivot phrasing.

**Fix**: pivot injection is now gated on the planned answer type —
`jd_fit_answer` / `gap_analysis_answer` / `behavioral_interview_answer` only.

### 3. Background meeting-end work contended with manual answers
After meeting end: `generateMeetingSummary` (LLM, up to 60s) + RAG chunking +
the embedding queue drain — all on the main process, all interleaving
**synchronous better-sqlite3 writes** with the user's questions. This is the
"lags after ~50 questions" mechanism: the longer the session, the more
embedding/persistence backlog competes with answers.

**Fix**: new `electron/services/ForegroundGate.ts` — manual/WTA answers mark
themselves in-flight; `EmbeddingPipeline.processQueue` and
`LiveRAGIndexer.tick` now `await ForegroundGate.waitUntilIdle()` between items.
Background work pauses during answers and resumes between them. (Gate is
advisory, auto-expires in 60s, capped wait 30s so background always progresses.)

### 4. Prompt bloat (25–32k chars)
Contributors measured in the trace: knowledge system-prompt injection +
CORE_IDENTITY/EXECUTION_CONTRACT + rolling 100s context + dossier + salary +
gap pivots + mock-question hints + culture mappings + custom notes.
Fixes #1/#2 remove the dossier and pivot blocks from answers that don't need
them. (The earlier PI v3 work also pinned mode customContext at 1,200 chars and
made reference retrieval index-backed.)

## Sequential-stress verification

See `MANUAL_SEQUENTIAL_STRESS_REPORT.md` (generated by
`npm run benchmark:manual-sequential-stress`) — 216 prompts in one process/
session across 18 sets (16=lecture, 17=sales, 18=looking-for-work), with
event-loop delay + heap tracking and a synthetic background-drain loop that
proves the ForegroundGate pauses background work during answers.

Gates enforced by the runner (exit 1 on failure):
- p95 first-useful < 2500ms · p99 < 3500ms
- provider_timeout = 0
- heap growth < 20% over the run · event-loop p95 < 250ms
